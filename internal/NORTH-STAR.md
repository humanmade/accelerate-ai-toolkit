# North Star

Accelerate AI Toolkit should become the best conversational operating layer for a non-technical marketer running a WordPress site with Accelerate.

Its job is not to expose internals. Its job is to turn site data into clear priorities, safe actions, and better decisions. If a marketer can describe a problem in one sentence, the toolkit should know how to help.

## Who It Is For

This project is for:

- marketers who want answers in plain English
- site owners who want to know what matters now
- operators who want recommendations they can act on immediately
- teams who want safe help with testing, personalization, campaigns, and content decisions

This project is not for:

- developers looking for raw platform access
- people who want a generic MCP demo
- users who prefer inspecting schemas over getting decisions made

## What Great Looks Like

Great work on this project does seven things:

- turns analytics into priorities, not just reports
- keeps the UX workflow-shaped, not wrapper-shaped
- speaks like a strong marketing operator, not a toolchain
- uses real site data for every meaningful recommendation
- makes changes safely, with explicit confirmation before mutations
- gets more useful as it learns the patterns of a specific site
- builds on generic, reusable platform primitives where possible, not bespoke upstream hooks that only this toolkit can use

It should also look trustworthy at first glance: clear naming, clean structure, consistent docs, and a level of polish that makes inclusion in Claude and Codex directories feel obvious rather than arguable.

## What We Will Not Become

We will not become:

- a developer toolkit front-end
- a 1:1 wrapper over every Accelerate capability
- a clone of the WordPress admin
- a generic assistant for every WordPress task
- a bloated catalog of thin commands and low-value skills
- a project that requires custom upstream behavior for every major feature

If a new feature makes the project feel more technical, more scattered, or more like an internal console, it is probably the wrong feature.

## Product Ambition

The ambition is bigger than "AI access to analytics."

We want this toolkit to own the loop from:

1. what is happening
2. what matters
3. what should we do next
4. can you help me do it safely
5. did it work

Over time, that should make the toolkit feel less like a reporting layer and more like a growth operating system for a real site.

## Ecosystem Standard

This repo should be directory-ready.

That means:

- presentation should be clear, credible, and easy to understand in one pass
- manifests, docs, and repo structure should feel complete and intentional
- wording should signal safety, usefulness, and product maturity
- shared skills and per-agent packaging should stay organised and coherent
- there should be no obvious drift between what we claim, what we ship, and what reviewers will find
- major feature, packaging, and wording changes should be checked against current official Claude and Codex guidance, not just our memory of it
- major features should align with open, legible data contracts and platform conventions, so the toolkit can work with future agent systems, not just the first one we build around

## Standard

Every new skill, command, doc, and manifest should make the toolkit more useful to a marketer making decisions, more trustworthy to ecosystem reviewers evaluating inclusion, and more aligned with current official platform expectations.
